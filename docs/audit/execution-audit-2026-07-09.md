# Telebot Execution Audit — Findings & Remediation

**Date:** 2026-07-09
**Method:** Multi-agent audit — 5 domain auditors (execution/TP-SL, auth/CSRF, API-500, industry standards, data consistency) read the codebase in parallel; all 18 critical/high findings were then independently re-verified by adversarial agents instructed to refute them against the actual code. **Zero findings were refuted.** Every finding cites the exact lines that prove it.

**Verdict:** 6 critical + 7 high verified findings (deduplicated), ~20 medium/low. All three operator-reported symptoms fully root-caused:

| # | Symptom | Root cause | Where |
|---|---------|-----------|-------|
| 1 | Trade executes but TP never attaches → runs to SL | Four independent mechanisms, all made permanent by the watchdog sibling exclusion | §1 |
| 2 | Second device gets "csrf token invalid" | CSRF cookie is browser-session-lifetime while auth session lives 30 days; SPA never reseeds | §2 |
| 3 | 500 "unable to get…" on active-trade details | asyncpg returns JSONB as `str`; drilldown calls `.get()` on it | §3 |

## 1. Symptom 1 — trade executes but TP never attaches, position rides to SL

This is not one bug. Four independent, verified mechanisms all end in the same state: a live position with `tp=0` that no background process will ever repair. **The common thread:** the orphan protective-TP watchdog's sibling exclusion (`db.py:1134-1138`) — as soon as a signal has stage ≥ 2 rows, the watchdog is *permanently* forbidden from touching stage 1, so any failure in the one-shot TP attach path is terminal.

### 1.1 [CRITICAL] Correlated follow-up bands persisted without `signal_sl` / `signal_tp`

**Location:** `trade_manager.py:488-503` → `executor.py:1026-1041`

`_handle_correlated_followup` builds the stage 2..N band rows without the `signal_sl`/`signal_tp` keys, so `db.create_staged_entries` inserts NULL for both (`db.py:1043` uses `r.get()`). The direct-zone path (`trade_manager.py:710-711`) and text-only path (`329-330`) both set them — this insert site was simply missed; the EXEC2-01 comment at `executor.py:616` even claims the row "carries the follow-up's real SL/TP".

Bands in-zone at arrival fire with the real TP, but any band that **arms** and later fires through the zone watcher reads the NULLs: it opens with `tp=0.0` (`trade_manager.py:971`, `tp=jittered_tp or 0.0`) and a default-pips SL measured off the **band edge** instead of the signal's SL (`executor.py:1009-1016, 1028`).

Second consequence: the same NULLs make the TP/SL-reached cascade fall back to the orphan's signals row (logged with `sl=0/tp=0`), so `executor.py:644` skips the cascade and **remaining stages keep firing after price already hit the real TP or SL**.

**Fix:** Add `"signal_sl": signal.sl, "signal_tp": signal.target_tp` to the rows dict, mirroring `trade_manager.py:710-711`. On pairing, also update the orphan's signal targets so the cascade fallback works.

### 1.2 [CRITICAL] Follow-up without a TP sends `tp=0.0` — MT5 interprets that as "remove the TP"

**Location:** `trade_manager.py:418-425` → `mt5-rest-server/server.py:372`

When the follow-up has no numeric TP (e.g. "TP: open"), `new_tp` stays `0.0` and is sent in the SLTP modify anyway. The bridge treats `0.0` as an explicit value (`"tp": req.tp if req.tp is not None else pos.tp` — `0.0 is not None`), so any existing TP is erased. Because the pairing just created sibling band rows, the watchdog exclusion (§1.1) guarantees no protective TP is ever re-attached.

Verifier refinement: erasing a *watchdog-attached* TP requires a timing race (the correlator pairs only within the window the watchdog waits out); the dominant reachable paths are (a) a no-TP follow-up whose sibling rows permanently block the watchdog from protecting a TP-less stage-1, and (b) wiping a TP set by a MODIFY_TP signal or manual terminal edit within the window. Identical end state either way.

**Fix:** `new_tp = calculate_tp_with_jitter(...) if signal.target_tp else None` — the server's `is not None` guard then preserves the existing TP. Broaden the watchdog to any bot-owned position with `tp==0` older than the correlation window, not only sibling-free stage-1s.

### 1.3 [CRITICAL] Stage-1 TP attach is single-shot with zero post-trade verification

**Location:** `trade_manager.py:423-472`, `mt5-rest-server/server.py:379-380`

The *only* mechanism that attaches the follow-up's TP to an already-open text-only position is one `modify_position` call. Broker rejects (retcode 10016 invalid-stops — likely exactly when price has run toward the TP — or 10025 no-changes) become HTTP 400 and get **zero** retries (`mt5_connector.py:603-606` returns immediately); only transport timeouts get 3 attempts. The failure is logged and appended as `stage1_align_failed` — which `notifier.py:39-69` has no case for, so it is **silently omitted from the Telegram execution message**.

No retry queue, no periodic re-check, no verification that the live position actually carries the requested SL/TP. One failed modify → position unprotected forever (sibling rows block the watchdog). The modify also bundles SL+TP in one `TRADE_ACTION_SLTP` request, so a TP rejected for stops-level discards the SL update too.

Scope note from verification: if `max_stages == 1` no sibling rows are created and the watchdog would eventually attach a protective TP — the permanent-loss path requires the normal `max_stages >= 2` configuration.

**Fix:** Persist the desired SL/TP onto the stage row at pairing time, then add a reconciliation sweep to the existing 10s zone-watch tick: any filled stage whose live position deviates from persisted targets gets a re-issued modify with bounded retries and an operator alert after N failures. Split SL and TP modifies (or fall back to SL-only) when TP is rejected. Add a `stage1_align_failed` case to the notifier.

### 1.4 [CRITICAL multi-account / latent single-account] Correlator pairs follow-ups to the wrong account's signal

**Location:** `executor.py:107-157`, `signal_correlator.py:69-79`

Each account runs in a temp per-account TradeManager, so one text-only Telegram message logs N distinct `signal_id`s (`trade_manager.py:269-276` runs once per account) and registers N orphans in the **shared** correlator — keyed only by `(symbol, direction)`, popped LIFO. Accounts are shuffled independently for the text-only and follow-up messages (`executor.py:108`), so a fully correct pairing requires the second shuffle to be the exact reverse of the first: probability 1/n! (50% failure with just 2 accounts).

A mispair is **symmetric** — both accounts silently skip stage-1 alignment (`trade_manager.py:407-414, 473-477` — debug-level log only), both are excluded from the watchdog (the sibling NOT EXISTS at `db.py:1134-1138` has no `account_name` predicate), the D-16 cascade (`executor.py:704-732` via account-blind `get_stage_by_comment`) wrongly cancels the other account's armed bands, and the duplicate-direction check (`trade_manager.py:829-835`) blocks band fills.

Reverting to one shared TradeManager doesn't fix it either: all accounts would share one signal_id and stage rows would collide on `UNIQUE(mt5_comment)`.

**Caveat:** the checked-in `accounts.json` has one enabled account, where this cannot fire — but it makes the advertised multi-account mode unsafe to enable, and `accounts.example.json` ships two.

**Fix:** Give one Telegram message one canonical `signal_id`: log the signal once in `Executor.execute_signal` before the fan-out and pass it down. Key the correlator by `(account, symbol, direction)` as defense-in-depth, and add `account_name` predicates to the sibling NOT EXISTS and `get_stage_by_comment`.

### Which one is biting you today?

On a single-account deployment, **1.1 is the most probable live cause** (any signal whose later stages fire via the zone watcher opens TP-less), with 1.2 and 1.3 covering follow-up-driven cases. All three are cheap fixes. 1.4 becomes mandatory before ever enabling a second account.

## 2. Symptom 2 — laptop + phone: one device works, the other gets "csrf token invalid"

The backend fully supports concurrent devices — there is no global token, no rotation, no server-side session singleton. Sessions are stateless signed cookies (`dashboard.py:179-187`) and the CSRF check is a stateless per-request double-submit compare (`api/deps.py:62-66`). The bug is a cookie lifecycle mismatch with no recovery path.

### 2.1 [HIGH] CSRF cookie dies with the browser; session cookie lives 30 days; SPA never reseeds

**Location:** `api/auth.py:80-87, 169-176` · `dashboard.py:183` · `frontend/src/auth/LoginView.tsx:42`

The `telebot_csrf` cookie is set **without** `max_age`/`expires` — a non-persistent browser-session cookie — while `telebot_session` persists 30 rolling days. When one device's browser restarts (or the mobile OS evicts session cookies, which iOS/Android do aggressively for backgrounded browsers), that device still auto-authenticates via the persistent session cookie (`App.tsx:20-27` renders the shell straight off `/auth/me` 200) — but `document.cookie` no longer contains a CSRF token.

The SPA seeds CSRF **only** in the login view (`LoginView.tsx:42` is the sole `seedCsrf()` call site, verified by grep), which never mounts for an authenticated user. Every mutation then sends an empty `X-CSRF-Token` (`http.ts:51-53`, `readCookie` returns `""`) → 403 "CSRF token invalid" (`api/deps.py:63-66`), and the global error handler only auto-recovers from 401 (`queryClient.ts:43`). The device is stuck until manual re-login — while the device whose browser kept running works fine. The same defect also breaks a *single* device after any browser restart within the 30-day session window.

**Fix (three coordinated changes; any one resolves the symptom, do 1+2 for robustness):**
1. Backend: add `max_age=30*24*60*60` (matching the session cookie) to both CSRF `set_cookie` calls in `api/auth.py`.
2. Frontend boot reseed: after `/auth/me` succeeds, call `seedCsrf()` when `readCookie('telebot_csrf')` is empty.
3. Frontend self-heal: in `lib/http.ts`, on a 403 "CSRF token invalid", fetch `/auth/csrf` once and retry the request a single time.

**Secondary auth findings (not the symptom):** non-revocable rolling 30-day sessions with client-only logout (medium); CSRF-exempt `GET /logout` enables cross-site forced logout (low); unprefixed/unbound double-submit cookie (low); login lockout keyed on unconditionally-trusted `X-Real-IP` (low). **Positives:** timing-safe compares throughout, argon2id with rate-limit ordered before the hash cost, config-driven Secure flag, correct `httponly=False` on the double-submit cookie.

## 3. Symptom 3 — 500 "unable to get…" on active-trade details

### 3.1 [HIGH] Drilldown crashes on every staged trade: JSONB arrives as `str`, code calls `.get()` on it

**Location:** `db.py:1284, 1291` · `api/positions.py:62`

The pool is created with plain `asyncpg.create_pool` (`db.py:60-65`) and `set_type_codec` appears nowhere in the repo, so asyncpg returns JSONB columns as raw JSON **strings** (written with `json.dumps` at `db.py:1039`). `get_position_drilldown` does `settings = stage["snapshot_settings"] or {}` — a non-empty JSON string is truthy — then `settings.get("default_sl_pips")` → `AttributeError: 'str' object has no attribute 'get'`.

The codebase itself proves the runtime type: `executor.py:898` and `dashboard.py:246` both `isinstance`-str-guard this exact column; `get_position_drilldown` is the one un-guarded reader.

It fires for precisely the trades with `staged_entries` rows joined on ticket (`db.py:1278`) — every staged/zone fill, since `trade_manager.py:1003/1013` writes the same `result.ticket` to both tables — while plain single-stage trades take the crash-free fallback at `db.py:1296-1305`, which is why only *some* details 500. Since `api/errors.py:83-88` registers handlers only for `HTTPException` and `RequestValidationError`, the `AttributeError` escapes as a bare un-enveloped 500 and the SPA shows its generic failure text — the reported "unable to get…".

**Fix (best):** register the codec once at pool init:
```python
asyncpg.create_pool(..., init=lambda conn: conn.set_type_codec(
    'jsonb', encoder=json.dumps, decoder=json.loads, schema='pg_catalog'))
```
then delete the scattered isinstance guards. **Minimal:** mirror `executor.py:898-904` in `get_position_drilldown`. Also add a catch-all exception handler so unexpected errors return the standard error envelope.

**Related drilldown polish:** the payload omits every `*_display` twin the SPA renders (fill history shows em-dashes); `sl_at_fill` mixes units (pips for staged fills, absolute price for single-stage).

## 4. Other verified execution risks (money-losing paths beyond the symptoms)

### 4.1 [CRITICAL] Blind retry of `POST /api/v1/order` on ReadTimeout — duplicate market orders

**Location:** `mt5_connector.py:586-619`

`_request` retries every request 3× on `ConnectError`, `ConnectTimeout` **and `ReadTimeout`**, with no method/path discrimination. A ReadTimeout means the order may already have filled (the bridge runs MT5 calls synchronously on its event loop, so a 15s client timeout while the loop is busy is realistic) — the retry then submits a second identical market order: doubled exposure with the same comment.

- The D-25 idempotency probe (`trade_manager.py:938-958`) runs once **before** `open_order`, never between HTTP retries — and is staged-only; the unstaged path uses the constant comment `"telebot"` with no probe at all.
- Reconciliation can't detect the duplicate afterwards: `_sync_positions` (`executor.py:256-277`) collapses positions into a by-comment dict, so same-comment duplicates merge to one entry.
- If all 3 attempts time out, the trade is recorded **failed** while a real position sits on the broker untracked — and the reconnect reconcile only adopts stages in awaiting states (`db.py:1082`), never `failed` ones.
- The bridge has zero dedup: nothing checks `req.comment` before `order_send` (`server.py:253-355`).

**Fix:** Retry only GETs and connection-level failures (request provably never left). On ReadTimeout of an order POST, probe positions by unique comment before any resend and adopt the position if found; or make the bridge idempotent (return existing ticket for an already-executed comment). Give unstaged orders unique comments first.

### 4.2 [CRITICAL] Duplicate stage fire: at-arrival firing races the 10s zone-watch poll with no row claim

**Location:** `trade_manager.py:715` · `executor.py:533` · `db.py:1055-1062, 1097-1105`

Band rows are inserted `awaiting_zone` and then fired in the same coroutine, flipping to `filled` only after the order returns (`trade_manager.py:1002-1003`) — with multiple awaited network calls in between. The independent zone-watch loop polls `status='awaiting_zone'` every 10s with no `SELECT ... FOR UPDATE`, no claim status, no compare-and-set. A tick landing inside the fire window submits the same stage twice; the D-25 probe passes for both because the second call's positions fetch runs before the first order lands on the broker. `update_stage_status` has no status precondition — last-writer-wins, one broker position untracked. Result: **2× intended lot exposure.**

**Fix:** Atomic claim before submit — `UPDATE staged_entries SET status='firing' WHERE id=$1 AND status='awaiting_zone' RETURNING id` — in both fire paths; submit only on claim success. Add status preconditions to `update_stage_status` so terminal states (including kill-switch cancellations) can never be overwritten by in-flight fills.

### 4.3 [HIGH] REST bridge runs blocking MT5 C calls directly on the asyncio event loop

**Location:** `mt5-rest-server/server.py:70-86`

`_run` is `async def` but calls the MT5 function inline. While one `order_send`/`history_deals_get` blocks, **everything** — `/ping`, `/price`, `/positions` — freezes. This feeds the ReadTimeout retries in §4.1, and sustained blockage (~45s across ping retries, per verification) makes the heartbeat declare the connection dead and re-login — the exact trigger for the known retcode-10027 "AutoTrading disabled on account change" cascade.

**Fix:** Module-level `ThreadPoolExecutor(max_workers=1)`; `_run` becomes `await loop.run_in_executor(_mt5_executor, functools.partial(fn, *args, **kwargs))`. Single worker preserves MT5 call serialization (the lib is not thread-safe) while the loop stays responsive. Move paired `mt5.last_error()` calls into the same executor job.

### 4.4 [HIGH] Close path hardcodes `ORDER_FILLING_IOC` that the open path negotiates around

**Location:** `mt5-rest-server/server.py:415`

`create_order` negotiates `type_filling` from `symbol_info.filling_mode` (with a comment noting brokers like Vantage reject IOC for pending orders), but `close_position` hardcodes IOC. On any symbol where the open path fell back to FOK or RETURN, every bot-initiated close — including the **kill-switch flatten** — fails with retcode 10030: positions the bot can open but never close. Latent on symbols whose DEAL bitmask includes IOC; guaranteed failure where it doesn't.

**Fix:** Extract the negotiation into a helper used by both paths; optionally retry a close once with RETURN on 10030.

### 4.5 [HIGH] Kill-switch race: pause flag checked once, before up to (N−1)×5s of stagger

**Location:** `executor.py:104-134`

`execute_signal` reads `_trading_paused` only at entry, then iterates accounts with deliberate 1–5s stagger delays. Hit the kill switch mid-stagger and the in-flight signal keeps opening **new** positions on remaining accounts after `emergency_close` flattened everything — you believe you're flat while carrying live exposure. The zone-watch loop already has the correct D-21 mid-tick re-check (`executor.py:683-684, 760-761`); the primary path doesn't.

**Fix:** Re-check the flag after each stagger sleep, before `_execute_single_account`, and break with a "skipped: kill switch" result.

### 4.6 [HIGH] Dashboard full-close permanently records pnl = 0.0

**Location:** `api/actions.py:84` · `db.py:363`

The close route writes `update_trade_close(ticket, account, 0.0, result.price)` as a "placeholder" — but the history-sync loop only scans `status='opened'` rows (`get_open_trade_tickets_for_account`, `db.py:363`; skip at `executor.py:484-486`), so once the route flips the row to `closed`, the real deal profit is never backfilled. Every dashboard-closed trade shows $0 in `/history` and permanently corrupts every analytics aggregate (win_rate, profit_factor, net_pnl — `db.py:596-796` all read `trades.pnl`).

**Fix:** Don't mark closed from the route — leave the row in a status history-sync scans (e.g. `closing`, added to the candidate query) and let the sync record the authoritative `deal.profit`; or fetch the closing deal immediately after close and write its profit. (Writing floating `pos.profit` captured pre-close is close but not exact.)

### 4.7 [MEDIUM] `DONE_PARTIAL` treated as total failure; no retcode taxonomy; deviation hardcoded

**Location:** `mt5-rest-server/server.py:340`

A partial fill (retcode 10010) returns `success=False, ticket=0`: the bot logs a failed trade while a live partial position exists — no trades DB row (PNL/history-sync misses it), stage/risk accounting treats the filled volume as failed (potential over-exposure), invisible in the dashboard. (It does keep its SL/TP — `server.py:298-299` — and symbol-scan management signals still reach it.) Transient retcodes (10004 requote, 10021 price-off, 10006 reject) are never retried; terminal ones (10014, 10016, 10027) aren't distinguished; slippage is a hardcoded `deviation: 20`.

**Fix:** On 10010, return success with actual filled volume/ticket. Classify retcodes retryable-vs-terminal (bounded retries with fresh price for requotes; distinct error codes for terminal). Make deviation a per-request config value.

## 5. Industry-standards gap analysis

Benchmarked against professional order-execution practice. The staged-entry design has real strengths — pre-submit intent rows, comment-based idempotency probes, reconnect reconcile, kill-switch drain — but the gaps cluster exactly where money is lost.

| Standard practice | Telebot today | Gap |
|---|---|---|
| Post-trade verification | Fire-and-forget: nothing confirms SL/TP landed after order_send/modify | Root of symptom 1 (§1.3); watchdog covers only sibling-free stage-1s |
| Idempotent order submission | Blind 3× retry incl. order POSTs; no client order-id enforced server-side; unstaged comment not unique | Double fills, untracked positions (§4.1) |
| Durable intent + reconciliation loop | Staged path writes intent rows; unstaged path has none; reconcile only on heartbeat reconnect, never at startup | Crash leaves phantom armed stages and orphan broker positions |
| Retryable vs terminal error taxonomy | Any retcode ≠ DONE → failure; transport errors retried indiscriminately | Requotes never retried; partial fills discarded (§4.7) |
| Kill-switch semantics | Checked once per signal; close path can't fill on FOK/RETURN symbols | Exposure after flatten (§4.5); flatten can fail entirely (§4.4) |
| Concurrency control on shared state | No row locks, no compare-and-set, no status preconditions in the stage state machine | Duplicate fires, resurrection of cancelled stages (§4.2) |
| Non-blocking broker gateway | MT5 C calls inline on the event loop | Self-inflicted timeouts, false reconnects, the 10027 cascade (§4.3) |
| Precision handling | Floats and hardcoded rounding; no broker volume_step/digits awareness | Occasional broker rejects; fixed-lot stage floor can exceed configured total |

# Appendix A — full findings with evidence and adversarial verdicts

## A.1 Domain summaries

### Domain: exec-tp-sl

The TP/SL attachment symptom is real and has two independent, provable root causes plus a fragile single-shot attach path. (1) CRITICAL: _handle_correlated_followup creates band rows WITHOUT signal_sl/signal_tp (trade_manager.py:488-503), so every stage that arms and later fires via the executor's zone watch opens with tp=0.0 and a default-pips SL off the band edge (executor.py:1026-1041 -> trade_manager.py:971); the orphan protective-TP watchdog explicitly excludes stages with siblings (db.py:1134-1138), so nothing ever attaches a TP and the position rides to SL — the same NULLs also neuter the tp_reached/sl_reached cascade (executor.py:644), so stages keep firing after the real TP is hit. (2) CRITICAL: in multi-account operation, each account logs its OWN signal_id per message (executor fans out per-account temp TradeManagers, executor.py:136/150-157; trade_manager.py:269) while the correlator pairs only by (symbol,direction) LIFO under two independent random shuffles (signal_correlator.py:69-79, executor.py:108) — cross-account mispairing (~50% with 2 accounts) makes the stage-1 SL/TP alignment silently no-op (trade_manager.py:407-414/473-477), permanently disqualifies the position from the protective watchdog, falsely cascades the other account's stages, and blocks band fills via the duplicate-direction check. (3) HIGH: even when pairing is correct, the TP attach is a single modify_position with no retry; a REST timeout or broker invalid-stops (10016) reject (server.py:379-380 -> HTTP 400 -> OrderResult failure) is logged and dropped, leaving the position TP-less forever.
