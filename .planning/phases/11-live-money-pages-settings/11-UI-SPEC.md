# Phase 11 — UI Design Contract

> Visual and interaction contract for the **Live-money Pages + Settings** phase.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** Overview (PAGE-05), Positions + 4 live-money actions (PAGE-06), two-step
> kill switch (PAGE-07), Settings with confirm-diff + audit + revert (PAGE-08) and the
> settings-UX bundle (SUX-01..04). The visual substrate (tokens, spacing, type, color,
> components) is **fully inherited from Phase 9 (`09-UI-SPEC.md`) and Phase 10**. Phase 11
> declares **NO new design tokens** — it specifies the live-money *interaction* contracts
> and copywriting those tokens render. Every value below is either inherited (cited) or a
> phase-specific interaction/copy contract.

## Spacing Scale

8-point design-token scale, inherited verbatim from Phase 9 (Tailwind v4 `0.25rem` step;
**all 7 tokens are multiples of 4** — this is the complete spacing-token contract):

| Token | Value | Usage in Phase 11 |
|-------|-------|-------------------|
| xs | 4px | Inline gaps (field label → help icon), badge internal gap |
| sm | 8px | Compact control spacing (input → inline help text), confirm-button gap |
| md | 16px | Default element spacing, card padding, settings field-to-field rhythm, table cell `px-4` |
| lg | 24px | Section padding, card inner padding, modal section gap |
| xl | 32px | Modal dialog padding, page-header-to-content separation |
| 2xl | 48px | Major section breaks (Overview sections: positions / pending-stages / kill-switch) |
| 3xl | 64px | Page-level spacing (rare) |

**Implementation-level Tailwind class choices (NOT design tokens — documented for the
executor, explicitly OUTSIDE the 7-token scale above):**

These two values are concrete component-level utility choices inherited from Phase 9 /
the legacy templates. They are **not** part of the spacing-token contract — the checker
should validate only the 7 tokens above. They are recorded here purely so the executor
reproduces the established density and tap-target behavior:

- **`py-3` (12px) table cell vertical padding** — a `DataTable` density choice inherited
  from Phase 10. It is a fixed component utility, not a reusable spacing token, and must
  not be treated as one.
- **`min-h-10` (40px) interactive touch-target minimum** — a component-level minimum
  height for nav rows, primary/destructive buttons, and mobile drawer links (matches
  Phase 9 / legacy `min-h-10`). The default shadcn button is `h-9` (36px); live-money
  destructive buttons (Close, CONFIRM CLOSE ALL, Confirm change) must use `size="default"`
  or larger to clear this minimum. This is an accessibility/ergonomics floor, not a
  spacing-scale step.

Inherited brand palette (60/30/10), oklch in `index.css`. **Phase 11 is the first phase to
USE `--destructive`** (themed-but-unused in Phase 9).

| Role | Value | Usage in Phase 11 |
|------|-------|-------------------|
| Dominant (60%) | `#0f0f1a` → `--background` `oklch(0.12 0.02 275)` | App background, main content surface |
| Secondary (30%) | `#1a1a2e` → `--card`/`--popover`/`--sidebar` `oklch(0.17 0.03 275)` | Cards, modal/dialog surface, sidebar, settings form card, kill-switch card |
| Elevated / muted | `#252542` → `--muted`/`--border`/`--input` `oklch(0.24 0.04 275)` | Borders, input borders, table header bg, muted surfaces |
| **Accent (10%)** | cyan `oklch(0.7 0.13 200)` → `--primary`/`--ring` | **Exclusive list below** |
| **Destructive** | red `oklch(0.62 0.21 25)` → `--destructive` | **Live-money destructive controls + danger signals** (see list) |
| Foreground | near-white `oklch(0.97 0 0)` | Body text (WCAG AA on `#0f0f1a`) |
| Muted foreground | mid-grey | Disabled-nav labels, meta text, field help, "remaining after" readout label |

**Accent (cyan `--primary`) reserved EXCLUSIVELY for:**
1. The **active** nav-link indicator (Positions / Settings become live this phase).
2. The brand wordmark "Telebot".
3. Focus rings (`--ring`) on inputs / buttons / interactive elements.
4. **Non-destructive** primary CTAs: **Save SL/TP**, **Confirm change** (settings),
   **Resume Trading**. (These are the cyan `variant="default"` buttons.)

> Never use cyan for: destructive actions, status indicators, P&L coloring, body text, or
> disabled controls. Scarcity is the contract.

**Destructive (`--destructive` red) reserved EXCLUSIVELY for:**
1. The per-position **Close** button (`variant="destructive"`).
2. The partial-close **Close lots** button (`variant="destructive"`) and its inline-confirm.
3. The **Emergency Kill Switch** entry button + the **CONFIRM CLOSE ALL** confirm button.
4. The **TRADING PAUSED** banner (red text/border) + the kill-switch warning copy.
5. Validation-error text under settings fields (`text-destructive`, role="alert").
6. The `ErrorPanel` icon/border on read failure (inherited).

**Status / P&L semantics (independent of accent AND destructive):**
- **green** (`text-green-400`) = profit / LIVE / trading enabled.
- **red** (`text-red-400`) = loss / disabled. *(P&L red is the status `text-red-400`, distinct
  from the `--destructive` button fill — both red, different roles, inherited DataTable.)*
- **amber / muted** = DRY-RUN.

The **compounded-exposure footgun warning (SUX-02 / D-06)** renders as an **amber/warning
inline note** (not `--destructive` red — it is a caution, not an error, and must not collide
with field-validation red). Use a warning treatment (e.g. `text-amber-400` + `AlertTriangle`
lucide icon) distinct from both the cyan accent and the destructive red.

## Visual Hierarchy & Focal Point

**Primary screen:** Overview (PAGE-05) — the operator's default live-money landing surface.

**Primary focal point:** the open-positions table headline (its count of open positions +
the aggregate P&L readout in `--font-mono`, P&L colored green/red by sign). This is the
first thing the operator's eye should land on — "how many positions are live and am I up or
down right now."

**Secondary anchor:** the **TRADING PAUSED** banner (red `--destructive` text/border),
shown above the positions table whenever `trading-status` reports paused. When present it
escalates above the P&L readout because a paused engine is the most consequential state to
notice; when absent the open-positions headline is unrivaled as the focal point.

**Tertiary:** the pending-stages card and the **Emergency Kill Switch** entry button sit
below the fold of attention — reachable but deliberately not competing with the headline
count/P&L for first-glance attention.

## Interaction Contracts (live-money — the heart of this phase)

These are binding behavioral contracts the checker and executor enforce. They restate the
locked invariants as concrete UI behavior.

### Positions actions (PAGE-06)

| Contract | Behavior |
|----------|----------|
| **Server-confirmed only (SC#1)** | Close / Save SL/TP / Close lots update or clear the row **only** in TanStack `useMutation` `onSuccess`. **No optimistic clear.** A position never appears closed while still live at the broker. |
| **Disabled-while-pending** | Every destructive/mutation button is `disabled={mutation.isPending}` and shows a pending label (see Copywriting). |
| **Error keeps modal open** | On mutation error the Edit modal **stays open with typed values preserved**; an error toast surfaces. The modal closes only on confirmed success. |
| **Two independent submits in Edit modal (D-02)** | SL/TP modify and partial-close are **separate forms / separate buttons / separate CSRF calls / separate toasts** — never one combined Save. |
| **Inline two-click confirm (D-03)** | Per-position **Close** and partial-close **Close lots** replace the legacy `hx-confirm` browser dialog with an **in-app** guard: first click → "Confirm close #{ticket}? ✓ / ✕"; second click fires (disabled-while-pending); ✕ cancels (recoverable misclick). Rendering: in-place button morph **or** small popover (one only — executor's choice within D-03). |
| **Modal outside polling subtree (SC#3)** | Edit modal + drilldown render outside the 3s-polling subtree so a background refetch never clobbers typed SL/TP/lots. Drilldown must survive refetch (multi-open allowed per legacy — discretion). |

### Partial-close input model (PAGE-06 / D-04)

- Operator **types absolute lots to CLOSE** directly (exactly the API contract — **no percent,
  no slider**; the percent model is the "75% trap", Pitfall 3 — do not re-introduce it).
- UI shows **current volume** (`*_display`) + a live **"Remaining after: X.XX"** readout.
- zod-validate `0 < value < volume`, rounded to the symbol lot step.
- Each submit carries a **client-generated UUID `request_id`** (Phase 8 D-10): same id + same
  params → cached 200 replay; same id + different params → **409** (surface a typed error toast).

### Kill switch (PAGE-07)

- Two-step **preview → confirm**: `GET /api/v2/emergency/preview` renders the preview (count of
  positions/orders to be closed) → **CONFIRM CLOSE ALL** (`variant="destructive"`,
  disabled-while-pending) → `POST /api/v2/emergency/close`.
- **Resume**: `POST /api/v2/emergency/resume` (cyan primary), shown only while paused.
- The **TRADING PAUSED** banner (red) shows on Overview whenever `trading-status` reports paused.

### Settings (PAGE-08 / SUX-01..04)

- **Every change runs validate → confirm-diff → confirm (D-05)** — no dangerous-field allow-list.
  `POST /settings/{account}/validate` → `{valid, errors, diff, dry_run_text}` → confirm modal
  rendering the **diff + dry-run** → **Confirm change** → `POST /settings/{account}` (re-validates
  server-side).
- **Toasts (SUX-01):** sonner success on confirm; explicit **rejection** toast on validation
  failure; **revert confirmation** toast.
- **Footgun in BOTH places, mode-aware (D-06/D-07):** inline live-recomputed as the operator
  edits `risk_value` / `max_stages`, AND restated in the confirm-diff modal.
  - **percent mode** → exposure compounds: warn on `max_stages × risk_value`.
  - **fixed_lot mode** → `risk_value` is the **TOTAL across `max_stages`, NOT per-trade**
    (operator-confirmed 2026-05-01) — **do NOT multiply**; warn on the absolute total.
  The footgun number is a client *calc* off two bare numerics (Pitfall-5-safe).
- **zod caps mirror server (SUX-03):** mode-dependent + per-account `risk_value` caps derived
  from `api/settings.py` `_validate` + `store.effective` — never hardcoded.
- **Audit timeline + revert (PAGE-08):** revert is itself a CSRF mutation with a confirmation
  toast (`POST /settings/{account}/revert`).
