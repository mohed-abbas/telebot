POST /api/v2/settings/{account}/revert → MutationResult {ok,success}. Body {account}. Inverts ONLY the latest change (no audit_id). 404 "Nothing to revert".

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → settings values | Risk-shaping field values cross into validation then persistence |
| client zod caps → server caps | Client validation is UX-only; server re-validates authoritatively |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-12 | Elevation of Privilege | SettingsForm zod caps | mitigate | zod mirrors validate_settings_form (makeSettingsSchema) as defense-in-depth; the server confirm re-validates (422 if re-breached) — a bypassed client cap cannot persist out-of-cap settings. Backed by Phase 8 test_api_settings + test_settings_form |
| T-11-13 | Tampering | validate/confirm/revert | mitigate | All via api() → X-CSRF-Token echoed; rejected 403 without it (Phase 8 test_api_csrf) |
| T-11-14 | Information disclosure | footgun (both inline + confirm) | accept | Pure client calc off two bare numerics already on the page; Pitfall-5-safe; no server internals |
| T-11-15 | Integrity | validate 200-on-invalid handling | mitigate | Page branches on data.valid (Pitfall 7) so a cap failure is surfaced as a rejection toast and never silently confirmed |
| T-11-16 | Information disclosure | rejection/error toasts | mitigate | Toast copy from the typed server errors map / errorMessage() only — no raw exception leaked |
</threat_model>
